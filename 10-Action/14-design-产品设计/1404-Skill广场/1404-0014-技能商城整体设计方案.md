# 技能商城 0→1 整体设计方案

> 版本：v1.0
> 日期：2026-06-27
> 状态：正式版
> 参考来源：1401-00（产品定位）、1404（页面设计）、1401-03（发布流程与审批）、1403（视觉规范）
> 定位：技能商城从 0 到 1 的完整设计方案，覆盖技能市场列表页、四港精选详情页、发布表单、审批后台、安全扫描、通知机制，作为 UI/UX 交付和前端开发规格的双向依据。

## 1. 产品定位与用户场景

### 1.1 三个子模块定位

| 模块 | 功能定位 | 核心价值 |
|---|---|---|
| **技能市场** | 基于 SkillHub 全量技能的搜索发现，支持模糊搜索、分类筛选、卡片浏览 | 连接用户与技能，降低使用门槛 |
| **四港精选** | 已上线并通过审核的精选技能列表（运营视角），支持推荐位管理 | 品质背书，帮助用户发现高价值技能 |
| **发布 Skill** | 用户提交技能申请 → 安全扫描 → 管理员审批 → 上架 | 连接开发者与平台，构建技能生态闭环 |

### 1.2 核心用户

| 角色 | 能力范围 | 代表行为 |
|---|---|---|
| **普通用户** | 搜索/浏览技能市场、四港精选；安装/卸载技能；提交发布申请；查看自己提交的审批状态 | 技能消费者 + 初级贡献者 |
| **开发者用户** | 通过发布表单提交技能申请；修改重提被驳回的技能 | 技能贡献者 |
| **审批管理员** | 审批/拒绝技能申请；查看扫描结果；打回修改 | 平台内容守卫者 |
| **平台管理员** | 上架/下架技能；配置分类；推荐位管理；用户权限管理 | 平台运营者 |

### 1.3 设计原则

- **信息密度适中**：卡片列表兼顾预览效率与信息完整度，无需跳转即可获取技能核心信息
- **视觉引导清晰**：分类标签色系区分技能（绿 `#22C55E`）与工具（青 `#00AAEE`）；Tab 蓝色下划线激活态
- **操作即时反馈**：安装按钮状态切换流畅（idle → loading → installed/error）；提交后实时展示状态时间线
- **安全优先**：所有用户提交内容必须经过三阶段安全扫描 + 人工审批后方可上线

## 8. 安全扫描设计

> 扫描服务已确认接入 **Cisco SafeX**（开源，有 API），非自建 DLP。

### 7.1 三项检测逻辑

| 检测项 | 范围 | 方式 | 阈值策略 |
|---|---|---|---|
| **敏感词检测** | 全部字段（名称/简介/功能/场景/提示词） | Cisco SafeX API + 关键词规则库（兜底） | `risk_level > 2` 判定失败 |
| **恶意代码检测** | 示例提示词（主要）、核心功能说明 | 正则规则引擎 + AST 解析（自建） | 命中任意一条规则即失败 |
| **提示词注入检测** | 示例提示词 | 规则引擎 + LLM API 二次判断（双轨） | 高风险规则直接失败；中风险 LLM 判断 |

**恶意代码检测规则示例**：

| 类型 | 规则 | 描述 |
|---|---|---|
| 代码执行 | `eval(`、`exec(`、`os.system`、`subprocess` | 检测到系统指令 |
| 文件路径 | `/etc/passwd`、`C:\Windows`、`%APPDATA%` | 检测到敏感路径 |
| 网络请求 | `requests.get`、`curl`、`http://` | 检测到外部网络调用 |
| 越狱指令 | `ignore previous instructions` | 尝试绕过安全限制 |

**提示词注入检测规则示例**：

| 风险级别 | 模式 | 描述 |
|---|---|---|
| 高 | `忽略上文`、`disregard all previous` | 指令覆盖攻击 |
| 高 | `你不再是 AI` | 身份否认指令 |
| 中 | `你现在是xxx` | 角色劫持指令 |
| 中 | 长 Base64 字符串 | 编码混淆攻击 |
| 中 | URL 编码内容 | 编码绕过 |

### 7.2 异步扫描架构

```
用户提交表单
      │
      ▼
POST /api/v1/skills/submissions
      │
      ▼
后端写入 DB（status=PENDING_SCAN）
      │
      ▼
发布 ScanJob 到 Redis Stream
      │  非阻塞（< 100ms 返回前端）
      ▼
前端轮询 GET /api/v1/skills/submissions/{id} 或 WebSocket 推送
      │
      ▼
┌──────────────────────────────────┐
│        扫描 Worker 服务           │
│  1. 消费 ScanJob 消息            │
│  2. 调用 Cisco SafeX API         │
│  3. 执行恶意代码检测（正则）       │
│  4. 执行提示词注入检测（规则+LLM） │
│  5. 写入 skill_scan_result 表    │
│  6. 调用回调 API 更新状态         │
└──────────────────────────────────┘
```

- 消息队列：MVP 使用 **Redis Stream**（零额外基础设施成本）
- 三项检测并行执行（`asyncio.gather`）
- SafeX API 超时：降级放过，不阻塞流程

### 7.3 扫描失败文案

| 失败类型 | 用户可见文案（友好版） |
|---|---|
| 敏感词-政治 | 您的技能内容包含不适合发布的关键词，请修改后重新提交。 |
| 敏感词-广告 | 您的技能内容包含广告或引流信息，不符合发布规范。 |
| 恶意代码-执行指令 | 示例提示词中包含系统指令，请调整描述方式后重新提交。 |
| 恶意代码-越狱 | 提示词内容疑似尝试绕过 AI 安全限制，请修改后重新提交。 |
| 提示词注入-角色劫持 | 提示词中的角色设定方式不符合规范，请调整后重新提交。 |
| 提示词注入-编码混淆 | 提示词中的特殊编码内容无法通过审核，请使用自然语言描述。 |

## 9. 通知机制

### 8.1 用户通知

| 触发时机 | 通知渠道 | 通知内容模板 |
|---|---|---|
| `SCAN_PASS` | 站内信 | 您的技能「{skill_name}」已通过安全扫描，正在排队等待审批。 |
| `SCAN_FAIL` | 站内信 + 推送 | 您的技能「{skill_name}」未通过安全扫描，请查看原因并修改。 |
| `PENDING_APPROVAL` | 站内信 | 您的技能「{skill_name}」已进入审批队列，预计 1-2 个工作日内完成审批。 |
| `APPROVED` | 站内信 + 推送 + 邮件 | 恭喜！您的技能「{skill_name}」已通过审批并成功上架！ |
| `REJECTED` | 站内信 + 推送 + 邮件 | 您的技能「{skill_name}」未通过审批，原因：{reject_reason} |
| `PUBLISHED` | 站内信 + 推送 | 您的技能「{skill_name}」已成功上架，可在「四港精选」中查看！ |

**通知优先级**：
- `SCAN_FAIL`、`REJECTED`、`PUBLISHED`、`APPROVED`：全渠道（站内信 + 推送 + 邮件）
- `SCAN_PASS`、`PENDING_APPROVAL`：站内信（低打扰）

### 8.2 管理员提醒

| 场景 | 提醒方式 | 触发条件 |
|---|---|---|
| 新申请待审批 | 管理员工作台红点 + 站内信 | `PENDING_APPROVAL` 队列 > 0 时，每天 9:00 提醒 |
| 待审批积压预警 | 站内信 + 邮件 | 同一申请等待超过 48 小时未处理 |
| 扫描失败率异常 | 邮件 | 单日扫描失败率 > 30% |
