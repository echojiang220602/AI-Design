## 1. 产品设计部分

## 1.1 发布表单设计

### 1.1.1 表单字段完整说明

| 序号 | 字段名 | 字段类型 | 必填 | 限制 | 校验规则 | 交互说明 |
|------|--------|----------|------|------|----------|----------|
| 1 | 技能名称 | 单行文本 | 是 | 最长 30 字符 | 非空，去首尾空格；不能与已上线技能完全同名（模糊查重） | 输入时实时字数计数，达到 28 字时输入框变橙色提示 |
| 2 | 分类 | 下拉选择 | 是 | 从后台配置的分类列表中选择 | 非空，必须为有效分类 ID | 搜索式下拉（支持拼音首字母搜索分类） |
| 3 | 一句话简介 | 多行文本（3行） | 是 | 最长 50 字符 | 非空；不能含纯标点或纯数字 | 实时字数计数，建议 20-40 字为佳 |
| 4 | 核心功能说明 | 多行文本（6行） | 是 | 最长 200 字符 | 非空；至少包含 2 个不同的关键词 | 实时字数计数，帮助用户凝练描述 |
| 5 | 使用场景 | 多行文本（4行） | 是 | 最长 200 字符 | 非空；描述应具体，可包含行业或岗位关键词 | 实时字数计数 |
| 6 | 示例提示词 | 多行文本（10行，支持代码块渲染） | 是 | 最少 20 字符，最多 2000 字符 | 非空；不支持纯 Markdown 特殊字符开头（`#` 标题格式禁止）；提示词中如包含变量占位符（如 `{name}`、`[file]`）需用反引号包裹 | 支持粘贴，带格式提示标签（自动识别 Markdown 代码块语法） |

**字段关联规则**：
- 分类选择影响「使用场景」的占位提示文案（如选"客服"，场景 placeholder 变为"描述该技能在客服场景下的典型应用"）
- 示例提示词字数统计区分为「有效字数」和「含变量占位符字数」两类展示

### 1.1.2 表单交互规范

**实时校验策略**：

| 触发时机 | 校验内容 | 提示形式 | 颜色 |
|----------|----------|----------|------|
| 失焦（blur）时 | 非空、格式基础校验 | 输入框下方红色文字 | `#EF4444` |
| 输入中（字数接近上限） | 字数 80% 时 | 输入框边框变橙色 | `#F59E0B` |
| 提交前（submit） | 全量校验 | 弹窗顶部红色提示条 + 对应字段高亮 | `#EF4444` |
| 提交时 | 唯一性校验（技能名称） | 同上 | `#EF4444` |

**错误提示文案规范**：
- 非空：`请填写 [字段名]`
- 超出字数：`[字段名] 不能超过 N 个字符，当前已输入 X 个`
- 格式错误：`示例提示词 不能以 # 开头，请修改后再提交`
- 名称重复：`该技能名称已被占用，请换一个名称`

**提交确认交互**：

```
┌─────────────────────────────────────────────┐
│              确认提交技能？                   │
│                                             │
│  技能名称：代码审查助手                        │
│  分类：研发工具                               │
│  一句话简介：自动审查 PR，发现潜在 bug...       │
│                                             │
│  提交后将进入安全扫描，预计 1-5 分钟完成        │
│                                             │
│         [取消]              [确认提交]        │
└─────────────────────────────────────────────┘
```

### 1.1.3 提示词粘贴区设计

**布局**：

```
┌──────────────────────────────────────────────────────┐
│  示例提示词                              [?]格式帮助  │
│──────────────────────────────────────────────────────│
│                                                      │
│  请粘贴或输入该技能的示例提示词                        │
│  （用户会看到并可直接复制使用）                        │
│                                                      │
│  ┌────────────────────────────────────────────────┐  │
│  │ 请帮我审查这个 PR，重点关注：                    │  │
│  │ 1. 潜在 bug                                     │  │
│  │ 2. 安全漏洞                                     │  │
│  │ 3. 代码规范问题                                 │  │
│  └────────────────────────────────────────────────┘  │
│                                                      │
│  字数：156 / 2000 字符              ✓ 已检测到代码块  │
│                                                      │
│  💡 小提示：提示词越具体，AI 理解越准确，使用效果越好   │
└──────────────────────────────────────────────────────┘
```

**格式支持**：
- 自动识别 Markdown 代码块（` ``` ` 包裹内容）并高亮渲染预览
- 支持变量占位符 `{name}`、`[file]`、`{{variable}}` 的识别和样式标注
- 粘贴时自动去除富文本格式（粘贴 HTML 时只保留纯文本）

### 1.1.4 多步骤 vs 单页面选择

**推荐方案：单页面居中显示（MVP），单页面宽度为整个页面的 50%**

| 方案 | 结论 |
|------|------|
| 多步骤引导式（3步向导） | 不推荐，发布是低频操作，用户愿接受一定复杂度，但操作路径长 |
| 单页面平铺 | **MVP 推荐**，信息一目了然，可随时修改 |
| 单页面 + 分区折叠 | V2 升级方向，支持渐进式展开 |

## 1.2 提交后流程设计

### 1.2.1 完整状态流转

```
                              ┌────────────────────────────┐
                              │                            │
                              ▼                            │
┌──────┐   submit   ┌──────────────┐   scan done   ┌────────────┐
│ DRAFT │ ─────────→│ PENDING_SCAN │ ────────────→│ SCAN_PASS  │
└──────┘            └──────────────┘              └────────────┘
                          │                            │
                          │ scan fail                  │ auto
                          ▼                            ▼
                   ┌──────────────┐           ┌─────────────────┐
                   │  SCAN_FAIL   │           │ PENDING_APPROVAL│
                   └──────────────┘           └─────────────────┘
                          │                            │
                          │ resubmit                   │ approve
                          └──────────────┐             ▼
                                       ┌──────────────┐    ┌──────────┐
                                       │ PENDING_SCAN │    │ APPROVED │
                                       └──────────────┘    └──────────┘
                                                                │
                          ┌────────────────────────────────────┘
                          │
                          ▼
                    ┌───────────┐
                    │ PUBLISHED │  ← 立即触发，全量用户可见
                    └───────────┘

              reject path (from PENDING_APPROVAL)
                          ┌──────────────┐
                          │  REJECTED    │
                          └──────────────┘
```

### 1.2.2 各状态节点用户可见信息

| 状态                 | 标签颜色   | 展示信息                       | 用户可操作        |
| ------------------ | ------ | -------------------------- | ------------ |
| `PENDING_SCAN`     | 蓝（处理中） | 提交时间；"正在排队等待安全扫描"          | 查看表单内容（只读）   |
| `SCANNING`         | 蓝（处理中） | 开始扫描时间                     | 查看表单内容（只读）   |
| `SCAN_PASS`        | 绿（成功）  | 扫描通过时间；提示"已通过安全扫描，等待管理员审批" | 查看表单内容（只读）   |
| `SCAN_FAIL`        | 红（失败）  | 扫描失败时间；**失败原因列表**（可点击展开详情） | 「修改重提」按钮     |
| `PENDING_APPROVAL` | 蓝（等待中） | 进入审批时间；队列位置（"当前排在第 N 位"）   | 查看表单内容（只读）   |
| `REJECTED`         | 红（拒绝）  | 拒绝时间；**拒绝理由**；审批人          | 「修改重提」按钮     |
| `PUBLISHED`        | 绿（上线）  | 上架时间；技能链接                  | 「查看技能」跳转四港精选 |

### 1.2.5 驳回后修改重提流程

```
用户看到 SCAN_FAIL / REJECTED
        │
        ▼
点击「修改信息并重新提交」
        │
        ▼
进入「修改模式」—— 表单恢复可编辑状态
（驳回前的填写内容保留，错误字段高亮标注）
        │
        ▼
修改完成后，点击「重新提交」
        │
        ▼
确认弹窗（显示本次修改点摘要）
        │
        ▼
状态重置为 PENDING_SCAN（保留原提交记录 + 新版本号）
        │
        ▼
进入正常扫描流程
```

**修改重提约束**：
- 同一技能最多允许修改重提 **5 次**（含初始提交），超过后需联系管理员
- 每次重提记录版本号（v1, v2, v3...）
- 驳回理由永久保留（可在技能详情页看到历次驳回记录）

### 1.3.3 审批操作设计

**通过操作**：
1. 点击「通过」→ 弹出审批备注输入框（非必填，最多 200 字）
2. 点击确认 → 状态变为 `APPROVED` → 自动触发上架 `PUBLISHED` → 发送通知

**拒绝操作**：
1. 点击「拒绝」→ **强制**填写拒绝理由（必填，10-200 字）
2. 拒绝理由模板（可多选或自定义）：
   - 功能描述不清，无法判断技能实际价值
   - 提示词质量不足，无法保证使用效果
   - 与已有技能高度重复
   - 涉及敏感内容，不适合上线
   - 其他（手动输入）
3. 点击确认 → 状态变为 `REJECTED`，理由通过站内信发送用户

**打回修改**（介于拒绝和通过之间）：
- 场景：内容基本合规，但有少量问题需要用户修正
- 操作：点击「打回修改」→ 强制填写打回原因 → 状态变为 `SCAN_FAIL` → 用户可修改内容并重新提交

### 1.3.4 审批历史记录

|------|------|
| 审批时间 | 精确到秒 |

## 1.4 通知机制

### 1.4.1 用户通知

| 触发时机 | 通知渠道 | 通知内容模板 |
|----------|----------|-------------|
| 状态变为 `SCAN_PASS` | 站内信 | 您的技能「{skill_name}」已通过安全扫描，正在排队等待审批。 |
| 状态变为 `SCAN_FAIL` | 站内信 + 推送 | 您的技能「{skill_name}」未通过安全扫描，请查看原因并修改。 |
| 状态变为 `PENDING_APPROVAL` | 站内信 | 您的技能「{skill_name}」已进入审批队列，预计 1-2 个工作日内完成审批。 |
| 状态变为 `APPROVED` | 站内信 + 推送 + 邮件 | 恭喜！您的技能「{skill_name}」已通过审批并成功上架！ |
| 状态变为 `REJECTED` | 站内信 + 推送 + 邮件 | 您的技能「{skill_name}」未通过审批，原因：{reject_reason} |
| 状态变为 `PUBLISHED` | 站内信 + 推送 | 您的技能「{skill_name}」已成功上架，可在「四港精选」中查看！ |

**通知优先级**：
- `SCAN_FAIL`、`REJECTED`、`PUBLISHED`、`APPROVED`：全渠道（站内信 + 推送 + 邮件）
- `SCAN_PASS`、`PENDING_APPROVAL`：站内信（低打扰）

### 1.4.2 管理员提醒

| 场景 | 提醒方式 | 触发条件 |
|------|----------|----------|
| 新申请待审批 | 管理员工作台红点 + 站内信 | `PENDING_APPROVAL` 队列 > 0 时，每天 9:00 提醒 |
| 待审批积压预警 | 站内信 + 邮件 | 同一申请等待超过 48 小时未处理 |
| 扫描失败率异常 | 邮件 | 单日扫描失败率 > 30% |

## 1.5 安全扫描设计

> ⚠️ **重要更新**：安全扫描已确认接入 **Cisco SafeX**（开源，有 API），非自建 DLP。

### 1.5.1 三项检测逻辑说明

**1. 敏感词检测**

| 项目 | 说明 |
|------|------|
| 检测范围 | 技能名称、一句话简介、核心功能说明、使用场景、示例提示词（全字段） |
| 检测方式 | Cisco SafeX API + 关键词规则库（兜底） |
| SafeX 能力 | 支持政治敏感词、暴力恐怖、色情低俗、广告引流等多类别检测 |
| 白名单机制 | 技术语境词汇（如"代码"、"脚本"）加入白名单避免误判 |
| 阈值策略 | SafeX 返回 risk_level > 2（中高风险）即判定失败 |

**2. 恶意代码检测**

| 项目 | 说明 |
|------|------|
| 检测范围 | 示例提示词（主要）、核心功能说明 |
| 检测方式 | 正则规则引擎 + AST 解析（自建） |
| 检测类型 | |
| → 代码执行类 | 包含 `eval(`、`exec(`、`os.system`、`subprocess` 等指令 |
| → 文件操作类 | 包含读写系统文件路径的指令（`/etc/`, `C:\`, `%APPDATA%`） |
| → 网络请求类 | 包含外部网络调用指令（`requests.get`、`http://`、`curl`） |
| → 越狱指令类 | 尝试绕过 AI 安全限制的提示词（如 "ignore previous instructions"） |
| 阈值策略 | 命中任意一条规则即判定失败 |

**3. 提示词注入检测（Prompt Injection）**

| 项目 | 说明 |
|------|------|
| 检测范围 | 示例提示词 |
| 检测方式 | 规则引擎 + LLM API 辅助判断（双轨） |
| 检测模式 | |
| → 角色扮演攻击 | 提示词中出现 "你现在是 xxx"、"你不再是 AI" 等角色劫持指令 |
| → 指令覆盖攻击 | "忽略上述指令"、"forget all previous"、"disregard" 等 |
| → 编码绕过 | Base64、URL 编码、Unicode 混淆的指令内容 |
| → 注入示例 | 在提示词中嵌入恶意示例，诱导 AI 学习错误行为 |
| 阈值策略 | 命中高风险规则直接失败；命中中风险规则进入 LLM 二次判断 |

### 1.5.2 扫描失败提示文案

| 失败类型 | 用户可见文案（友好版） |
|----------|----------------------|
| 敏感词-政治 | 您的技能内容包含不适合发布的关键词，请修改后重新提交。 |
| 敏感词-广告 | 您的技能内容包含广告或引流信息，不符合发布规范。 |
| 恶意代码-执行指令 | 示例提示词中包含系统指令，请调整描述方式后重新提交。 |
| 恶意代码-越狱 | 提示词内容疑似尝试绕过 AI 安全限制，请修改后重新提交。 |
| 提示词注入-角色劫持 | 提示词中的角色设定方式不符合规范，请调整后重新提交。 |
| 提示词注入-编码混淆 | 提示词中的特殊编码内容无法通过审核，请使用自然语言描述。 |

### 1.5.3 误判申诉机制

class SkillStatus(str, Enum):

    APPROVED          = "APPROVED"           # 审批通过（待上架）
    REJECTED          = "REJECTED"           # 审批拒绝
    PUBLISHED         = "PUBLISHED"          # 已上架（全量用户可见）

```python
# skill_state_machine.py
class SkillStateMachine:
    TRANSITIONS: dict[tuple, tuple] = {
        # (当前状态, 事件) → (目标状态, 触发动作列表)
        (SkillStatus.DRAFT, "SUBMIT"):              (SkillStatus.PENDING_SCAN,     ["enqueue_scan"]),
        (SkillStatus.PENDING_SCAN, "SCAN_DONE_PASS"): (SkillStatus.SCAN_PASS,      ["record_scan_result"]),
        (SkillStatus.PENDING_SCAN, "SCAN_DONE_FAIL"): (SkillStatus.SCAN_FAIL,      ["notify_scan_fail"]),
        (SkillStatus.SCANNING, "SCAN_DONE_PASS"):    (SkillStatus.SCAN_PASS,       ["record_scan_result"]),
        (SkillStatus.SCANNING, "SCAN_DONE_FAIL"):    (SkillStatus.SCAN_FAIL,       ["notify_scan_fail"]),
        (SkillStatus.SCAN_PASS, "AUTO"):             (SkillStatus.PENDING_APPROVAL, ["enqueue_audit", "notify_pending"]),
        (SkillStatus.SCAN_FAIL, "RESUBMIT"):         (SkillStatus.PENDING_SCAN,    ["increment_version"]),
        (SkillStatus.PENDING_APPROVAL, "APPROVE"):   (SkillStatus.APPROVED,        ["notify_approved"]),
        (SkillStatus.PENDING_APPROVAL, "REJECT"):    (SkillStatus.REJECTED,        ["notify_rejected"]),
        (SkillStatus.PENDING_APPROVAL, "REVISE"):    (SkillStatus.SCAN_FAIL,       ["notify_revise"]),
        (SkillStatus.APPROVED, "AUTO"):              (SkillStatus.PUBLISHED,       ["publish", "notify_published"]),
        (SkillStatus.REJECTED, "RESUBMIT"):          (SkillStatus.PENDING_SCAN,    ["increment_version"]),
    }

    @classmethod
    def transition(cls, current: SkillStatus, event: str) -> tuple[SkillStatus, list[str]]:
        key = (current, event)
        if key not in cls.TRANSITIONS:
            raise InvalidTransitionError(f"Cannot transition from {current} via {event}")
        return cls.TRANSITIONS[key]
```

| 事件名 | 触发时机 | 触发动作 |

### 2.3.2 三项检测代码实现

```python
# scan_service.py
import re
import httpx
from dataclasses import dataclass, field

@dataclass
class ScanResult:
    check_type: str
    passed: bool
    reason: str = ""
    matched_items: list = field(default_factory=list)

# ── 1. 敏感词检测：对接 Cisco SafeX ─────────────────────
SAFEX_API_URL = "http://<safex-host>:<port>/api/v1/check"
SAFEX_API_KEY = os.getenv("SAFEX_API_KEY")

async def check_sensitive(text: str) -> ScanResult:
    """基于 Cisco SafeX API 的敏感词检测"""
    # Step 1: SafeX API 调用
    try:
        async with httpx.AsyncClient(timeout=5.0) as client:
            resp = await client.post(
                SAFEX_API_URL,
                headers={"Authorization": f"Bearer {SAFEX_API_KEY}"},
                json={
                    "text": text,
                    "categories": ["politics", "violence", "porn", "ad", "fraud"]
                }
            )
            result = resp.json()
            if result.get("risk_level", 0) > 2:
                return ScanResult(
                    check_type="sensitive",
                    passed=False,
                    reason="内容审核未通过",
                    matched_items=[result.get("label", "")]
                )
    except (httpx.TimeoutException, httpx.RequestError):
        pass  # SafeX 不可用时跳过，不阻塞流程

    return ScanResult(check_type="sensitive", passed=True)


# ── 2. 恶意代码检测：正则规则引擎 ───────────────────────
MALWARE_PATTERNS = [
    # 代码执行
    (r'\beval\s*\(',                          "检测到 eval 指令"),
    (r'\bexec\s*\(',                          "检测到 exec 指令"),
    (r'\bos\.system\s*\(',                    "检测到系统命令执行"),
    (r'\bsubprocess\s*\.run\s*\(',            "检测到子进程执行"),
    (r'\b__import__\s*\(',                    "检测到动态导入"),
    # 文件路径类
    (r'/etc/(passwd|shadow|hosts)',            "检测到敏感系统路径"),
    (r'C:\\(?:Windows|System32)',              "检测到 Windows 系统路径"),
    (r'%APPDATA%|%TEMP%|%USERPROFILE%',        "检测到环境变量路径"),
    # 网络请求类
    (r'\brequests\.(get|post)\s*\(',           "检测到 HTTP 请求指令"),
    (r'\bcurl\s+',                             "检测到 curl 指令"),
    (r'http[s]?://(?!api\.)',                  "检测到外部网络请求"),
]

async def check_malware(text: str) -> ScanResult:
    """基于正则规则的恶意代码检测"""
    matched = []
    for pattern, description in MALWARE_PATTERNS:
        if re.search(pattern, text, re.IGNORECASE):
            matched.append(description)

    if matched:
        return ScanResult(
            check_type="malware",
            passed=False,
            reason="示例提示词包含疑似代码执行指令",
            matched_items=matched
        )
    return ScanResult(check_type="malware", passed=True)


# ── 3. 提示词注入检测：规则引擎 + LLM 二次判断 ──────────
INJECTION_PATTERNS = [
    (r'(忘掉|忘记|ignore|disregard)\s+(all\s+)?(previous|above)',
        "检测到指令覆盖攻击", "high"),
    (r'(你|you)\s+(不再|are\s+not?)\s+(a?n?\s+)?(AI|人工智能|语言模型)',
        "检测到身份否认指令", "high"),
    (r'你现在是(?!.*正在)',                     "检测到角色劫持指令", "medium"),
    (r'[A-Za-z0-9+/]{50,}={0,2}',             "检测到长 Base64 编码", "medium"),
    (r'%[0-9A-Fa-f]{2}%[0-9A-Fa-f]{2}',       "检测到 URL 编码内容", "medium"),
]

async def check_injection(text: str) -> ScanResult:
    """提示词注入检测：规则引擎 + LLM 二次判断"""
    matched = []
    high_risk = []

    for pattern, description, risk in INJECTION_PATTERNS:
        if re.search(pattern, text, re.IGNORECASE):
            matched.append(description)
            if risk == "high":
                high_risk.append(description)

    # 高风险直接失败
    if high_risk:
        return ScanResult(
            check_type="injection",
            passed=False,
            reason="提示词包含高风险注入指令",
            matched_items=high_risk
        )

    # 中风险交给 LLM 二次判断
    if matched:
        try:
            llm_result = await call_llm_judgment(text, matched)
            if llm_result.is_injection:
                return ScanResult(
                    check_type="injection",
                    passed=False,
                    reason="经智能检测，提示词存在注入风险",
                    matched_items=[llm_result.reason]
                )
        except httpx.TimeoutException:
            pass  # LLM 超时：降级保守策略

    return ScanResult(check_type="injection", passed=True)
```

### 2.3.3 扫描 Worker 实现

```python
# scan_worker.py
import asyncio
import httpx
from redis import asyncio as aioredis

async def scan_worker():
    redis = await aioredis.from_url("redis://localhost/1")
    stream_key = "skill:scan:jobs"

    while True:
        job = await redis.xread({stream_key: "0"}, count=1, block=5000)
        if not job:
            continue

        for stream_name, messages in job:
            for msg_id, fields in messages:
                submission_id = fields["submission_id"]
                text_to_scan = fields["text_to_scan"]

                # 更新状态为 SCANNING
                await db.update_status(submission_id, "SCANNING")

                t0 = asyncio.get_event_loop().time()

                # 三项检测并行执行
                sensitive_task = check_sensitive(text_to_scan)
                malware_task   = check_malware(text_to_scan)
                injection_task = check_injection(text_to_scan)

                results = await asyncio.gather(
                    sensitive_task, malware_task, injection_task,
                    return_exceptions=True
                )
                duration_ms = int((asyncio.get_event_loop().time() - t0) * 1000)

                scan_result = {
                    "sensitive_check": results[0] if not isinstance(results[0], Exception)
                        else {"error": str(results[0]), "passed": True},  # 异常时放过
                    "malware_check":   results[1] if not isinstance(results[1], Exception)
                        else {"error": str(results[1]), "passed": True},
                    "injection_check": results[2] if not isinstance(results[2], Exception)
                        else {"error": str(results[2]), "passed": True},
                }
                overall_passed = all(
                    not isinstance(r, Exception) and getattr(r, "passed", False)
                    for r in results
                )

                # 写入扫描结果
                await db.save_scan_result(submission_id, scan_result, overall_passed, duration_ms)

                # 更新状态并通知
                event = "SCAN_DONE_PASS" if overall_passed else "SCAN_DONE_FAIL"
                await db.transition_status(submission_id, event)

                await redis.xdel(stream_key, msg_id)
                print(f"Scan complete for {submission_id}: {'PASS' if overall_passed else 'FAIL'}")
```

# audit_reminder.py（每天 9:00 执行）
async def send_audit_reminder():
    pending_count = await db.fetch_val("""
        SELECT COUNT(*) FROM skill_submission
        WHERE status = 'PENDING_APPROVAL' AND deleted_at IS NULL
    """)

    if pending_count > 0:
        admin_users = await db.fetch_all("""
            SELECT id, email FROM users
            WHERE role IN ('AUDIT_ADMIN', 'PLATFORM_ADMIN') AND is_active = true
        """)
        for admin in admin_users:
            await send_notification(
                user_id=admin["id"],
                notif_type=NotificationType.AUDIT_REMINDER,
                payload={"pending_count": pending_count, "queue_url": "/admin/audit"},
                channels=["inapp", "email"]
            )
