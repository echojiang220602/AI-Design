## 一、产品定位

技能商城是 AI 数字员工平台的技能分发与发布平台，为用户提供技能「发现 → 使用 → 发布 → 管理」的全链路能力。连接 SkillHub 技能市场与四港精选能力，形成「发现—使用—贡献」的闭环生态。

## 二、产品架构

### 2.1 页面结构

技能商城入口位于左侧导航栏，默认为「技能市场」页面，内含三个 Tab：

| Tab 名称       | 功能定位                                |
| ------------ | ----------------------------------- |
| **技能市场**     | 基于 SkillHub 全量技能的搜索发现，支持模糊搜索、展示技能概述 |
| **四港精选**     | 已上线并通过审核的精选技能列表（运营视角）               |
| **发布 Skill** | 用户提交技能申请 → 安全扫描 → 管理员审批 → 上架        |

## 三、功能模块详设

### 3.1 技能市场（搜索发现）

#### 搜索能力

| 功能 | 描述 |
|---|---|
| 搜索方式 | 关键词模糊搜索，对接 SkillHub 全量技能数据 |
| 搜索字段 | 技能名称、核心功能、使用场景、提示词内容 |
| 过滤维度 | 按使用场景、按技能类型/分类 |
| 历史记录 | 保留最近 10 条搜索关键词 |
| 热门推荐 | 展示高调用量/高评分技能 |

> ⚠️ 当前无搜索 API，需新建。建议使用 **Elasticsearch** 或 **Milvus** 构建技能全文索引。

### 3.3 发布 Skill

#### 提交流程

```
步骤1：填写信息
    │
    ├── 技能名称（必填，最长 30 字）
    ├── 分类（必填，下拉选择）
    ├── 一句话简介（必填，最长 50 字）
    ├── 核心功能说明（必填，最长 200 字）
    ├── 使用场景（必填，最长 200 字）
    └── 示例提示词（必填，粘贴或输入）

步骤2：提交申请
    ↓
步骤3：自动化安全扫描
    ├─ 敏感词检测（基于 DLP/内容审核 API）
    ├─ 恶意代码检测
    └─ 提示词注入风险检测（Prompt Injection）

    ├─ 扫描通过 → 进入「待审批」队列
    └─ 扫描失败 → 打回「需修改」，显示具体原因

步骤4：管理员审批
    ├─ 审批通过 → 技能上架「四港精选」
    └─ 审批拒绝 → 附原因，打回修改

步骤5：上架完成
    └── 用户可在「四港精选」看到已发布的技能
```

#### 审批状态说明

| 状态 | 展示给申请人 |
|---|---|
| `待审核` | 提交成功，等待安全扫描 |
| `扫描中` | 安全扫描进行中 |
| `需修改` | 扫描/审核发现问题，附打回原因，可修改后重新提交 |
| `审批通过` | 审核通过，已上架 |
| `审批拒绝` | 未通过审核，附原因 |

#### 提交信息字段规范

| 字段 | 类型 | 必填 | 限制 |
|---|---|---|---|
| 技能名称 | 文本 | 是 | 最长 30 字 |
| 分类 | 下拉选择 | 是 | 如：客服、销售、分析、运营等 |
| 一句话简介 | 文本 | 是 | 最长 50 字 |
| 核心功能说明 | 文本 | 是 | 最长 200 字 |
| 使用场景 | 文本 | 是 | 最长 200 字 |
| 示例提示词 | 多行文本 | 是 | 最低 20 字 |

## 五、技术实现建议

### 5.1 搜索能力建设

当前无搜索 API，建议两条路径：

| 方案 | 技术选型 | 适用场景 |
|---|---|---|
| 方案 A | Elasticsearch 全文索引 | 结构化数据搜索，关键词匹配优先 |
| 方案 B | Milvus 向量数据库 | 语义相似度搜索，体验更智能 |

**推荐方案 A** 作为 MVP，理由：实现成本低、见效快、SkillHub 数据结构适合 ES 索引。

### 5.2 安全扫描

| 检测项 | 实现方式 |
|---|---|
| 敏感词检测 | 接入 DLP 内容审核 API 或关键词规则库 |
| 恶意代码检测 | 正则 + AST 解析检测代码注入 |
| 提示词注入检测 | 规则引擎识别角色扮演、越狱指令 |

### 5.3 审批流

建议引入**状态机**管理技能发布流程：

```
DRAFT → PENDING_SCAN → SCANNING → SCAN_PASS/SCAN_FAIL
                                          ↓
                              PENDING_APPROVAL → APPROVED/REJECTED
                                                      ↓
                                                   PUBLISHED
```

### 5.4 数据模型（概念）

```
Skill {
  id, name, category, summary,
  core_features, use_case, example_prompt,
  status: DRAFT|PENDING_SCAN|SCANNING|PENDING_APPROVAL|APPROVED|REJECTED|PUBLISHED,
  scan_result, reject_reason,
  version, created_by, created_at, updated_at
}
```
