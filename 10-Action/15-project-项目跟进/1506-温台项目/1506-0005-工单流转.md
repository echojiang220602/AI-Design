# 工单流转系统

## 一、工单类型

| 类型 | 说明 |
|------|------|
| **一访工单** | 首次拜访客户的工单 |
| **二访工单** | 二次跟进客户的工单 |

## 三、工单状态流转

```
待执行 → 执行中 → 待协助 → 协助中 → 协助完成 → 已完成
                                        ↓
                                    已终止
```

### 状态说明

| 状态 | 说明 |
|------|------|
| **待执行** | 工单已创建，等待业务员开始执行 |
| **执行中** | 业务员正在执行走访任务 |
| **待协助** | 执行中遇到问题，需要其他人员协助 |
| **协助中** | 协助人员正在处理 |
| **协助完成** | 协助任务完成，等待继续执行 |
| **已完成** | 整个工单流程结束 |
| **已终止** | 工单被取消或终止 |

---

## 四、工单操作

### 4.1 执行工单

业务员开始处理工单，状态从「待执行」变为「执行中」。

### 4.2 协助工单

当执行中遇到需要跨部门或其他人协助的场景时：

| 操作 | 触发条件 | 状态变化 |
|------|----------|----------|
| 发起协助 | 执行中需要他人帮助 | 执行中 → 待协助 |
| 接受协助 | 协助人员接单 | 待协助 → 协助中 |
| 完成协助 | 协助任务完成 | 协助中 → 协助完成 |
| 继续执行 | 业务员继续完成工单 | 协助完成 → 已完成 |

## 五、我的理解

### 业务逻辑

这是一个典型的**B端客户走访工单系统**，用于管理一线业务员拜访客户的全流程。

**核心场景**：
- 业务员创建工单（计划哪天去拜访哪个客户）
- 按计划执行走访
- 遇到问题可发起协助请求
- 完成后归档

### 设计亮点

1. **状态机设计清晰**：7个状态覆盖完整生命周期，状态流转有明确边界
2. **协助机制**：支持跨部门协作，解决了一线业务员能力边界问题
3. **终止路径**：除了正常完成外，提供终止路径，适应业务变化

### 可能的产品优化方向

| 方向 | 说明 |
|------|------|
| 与通话记录打通 | 执行工单时直接关联录音，转结构化报告 |
| 自动化状态推进 | 走访问卷完成后自动触发状态变更 |
| 协助能力库 | 建立公司内部技能/能力目录，快速匹配协助人 |
| 与AI数字人中台整合 | 客户历史工单自动聚合到企业画像 |

## 六、验收标准

### 功能验收

| 验收项 | 通过条件 |
|--------|----------|
| **工单创建** | 必填字段未填写时阻止提交，填写完整后成功创建并生成唯一工单号 |
| **状态流转** | 状态变更严格按「待执行→执行中→待协助→协助中→协助完成→已完成」路径执行，禁止跨状态跳转 |
| **协助流程** | 发起协助后原业务员无法单独完成工单；协助完成后自动通知原业务员 |
| **终止操作** | 任意状态下可终止，终止后工单进入只读状态，不可恢复 |

### 数据验收

| 验收项 | 通过条件 |
|--------|----------|
| **字段完整性** | 所有必填字段按工单类型正确校验，一访/二访分别校验关联项目字段 |
| **状态一致性** | 工单状态变更记录完整，支持追溯任意时刻的状态快照 |
| **关联数据** | 工单与关联项目、企业信息保持一致，删除关联数据时提示业务影响 |

### 异常验收

| 验收项 | 通过条件 |
|--------|----------|
| **并发冲突** | 同一工单被多人同时操作时，后者操作失败并提示「工单已被其他用户处理」 |
| **超时锁定** | 执行中状态超过设定的阈值（如48小时无操作），自动提醒业务员 |
| **权限隔离** | 非工单负责人无法操作工单状态，协助人员仅限查看和完成协助任务 |

### 性能验收

| 验收项 | 通过条件 |
|--------|----------|
| **列表加载** | 工单列表加载时间 ≤ 2秒（数据量 ≤ 1000条时） |
| **状态变更** | 状态变更操作响应时间 ≤ 500ms |
