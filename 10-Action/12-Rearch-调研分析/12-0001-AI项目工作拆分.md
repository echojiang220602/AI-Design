# AI 数字人中台 · 模块分类与拆分

> 基于 [[260706-AI数字人中台项目方案]] 的结构化分析

---

## 一、用户旅程模块（业务线）

模块边界跟着**导航（一级路由）走**，不跟着功能走。

| 模块 | 路由 | 价值定位 | 优先级 |
|------|------|---------|--------|
| **创建 Agent** | `/` | 入口钩子，两步表单 + 5 阶段 Loading | P0 |
| **我的 Agent** | `/my-agents/*` | 核心工作台（6 个子页面） | P0 |
| **使用指南** | `/guide` | 用户教育，降低上手成本 | P0 |
| **技能商城** | `/skills/*` | 扩展生态，分发技能 | P0 |
| **分享记录** | `/diary` | 留存复盘，情感连接 | P0 |

**依据**：方案 3.1 顶部导航明确给出 5 个一级导航项；3.2 用户流程图中"我的 Agent"是流程中枢节点；1.1 核心需求表给出 P0/P1/P2 优先级。

---

## 二、「我的 Agent」子模块（核心工作台）

6 个子页面本质都是**同一个 Agent 的不同操作维度**，按"对 Agent 的作用"分层而不是按"长得像不像"分层。

```
我的 Agent
├── 智能体（人格层）    → SOUL/USER/WORK/IDENTITY 文件 + 技能 + 定时任务 + 热力图
├── 文件夹（数据层）    → 文件树 + Monaco 编辑 + 上传下载
├── 终端（执行层）      → xterm.js 命令行（P1 占位）
├── 控制台（控制层）    → OpenClaw 集成（P1 占位）
├── 远程桌面（操作层）  → VNC/WebRTC（P2 占位）
└── 设置（配置层）      → Agent 配置持久化
```

**依据**：方案 1.1 核心需求表的 6 行 + 4.2 我的 Agent 子页表。按业务职责拆分（人格/数据/执行/控制/操作/配置），不是按视觉相似性。

## 四、分类依据优先级

所有分类判断的依据优先级（从高到低）：

1. **方案文档自述**：方案自己写的章节、表格、流程图
2. **全局规范**：CLAUDE.md 里的技术栈、目录、状态管理决策
3. **业务结构规律**：用户旅程、职责分层（人格/数据/执行/控制/操作/配置）
4. **技术债信号**：方案 6.2/6.3 作者自标的"待完善"
5. **一致性检查**：验收维度、命名规范、目录结构是否一致

## 五、关键洞察与建议

### 5.1 shrimp 模块需决策整合

**问题**：方案 6.2 自标技术债
> 存在 `src/pages/shrimp/` 和 `src/pages/Home/` 两套体系，需统一

**两种处理方案**：

- **方案 A（推荐）**：保留 shrimp 作为「我的 Agent」内部子模块（因带 Sidebar 布局，是工作台形态）
- **方案 B**：全部合并到 `pages/`，统一一套路由

### 5.2 技能商城应提升为一级模块

**依据**：方案 4.4 显示技能商城有 6 个子页面（商城首页 / 包详情 / 技能详情 / 市场 / 发布 / 审核），数量已超过"我的 Agent"的 6 个子页面，体量够独立。

### 5.3 验收 checklist 应模板化

**依据**：方案 1.3.1-1.3.13 共 13 个页面的验收表，**每页都用同一套 10 个维度**（方案 1.2 已定义）。重复 13 次 = 模板信号。

**动作**：抽出一个验收 checklist 模板，所有页面复用。

### 5.4 缺失的模块

| 缺失项 | 依据 |
|--------|------|
| **账号体系** | 1.3.1 创建 Agent 需要 Client ID/Secret，但全文未说明对接哪个账号系统 |
| **通知系统** | 1.3.10/1.3.11/1.3.12 提到 toast，但未设计统一通知中心 |
| **数据大盘** | 1.3.3 AgentPage 提到热力图，但整体层面没有 Dashboard |

## 六、分类边界对照

| 没有合并 | 依据 |
|---------|------|
| 没把"技能商城"放进"我的 Agent" | 3.1 顶部导航里它们都是并列的一级菜单 |
| 没把"分享记录"放进"智能体" | 1.1 需求表里它们是独立的页面行，不是子页面 |
| 没合并 `/skills`、`/diary`、`/my-agents/*` | 路由上是平级关系，不是嵌套关系 |

**反推**：若要合并三者，需要修改方案本身的导航和路由设计，而非简单分类调整。

---

## 七、项目本质总结

**「Agent 管理工作台」+ 「技能分发商城」+ 「个人日记」** 三件套

- **Agent 管理工作台**：核心（人格 + 数据 + 执行 + 控制 + 操作 + 配置 6 维度）
- **技能分发商城**：生态（6 个子页面，分发 + 审核 + 发布）
- **个人日记**：留存（时间线 + 详情 + 上传）

以 **Agent 为核心**、**技能为生态**、**日记为留存**。
